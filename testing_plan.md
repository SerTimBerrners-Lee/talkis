# Тестирование стилей обработки Talkis

## Конфигурация тестов

| Параметр | Значение |
|---|---|
| **Язык** | `ru` (основной), `en` (дополнительно) |
| **Стили** | Classic, Business, Tech |
| **Модель STT** | gpt-4o-transcribe |
| **Модель LLM** | gpt-4o-mini |

---

## 1. 🎯 Классический стиль (`classic`)

> **Цель:** Минимальная обработка. Текст максимально близок к тому, что сказал человек. Удаляются только явные филлеры и опечатки STT.

| # | Что говорим | Ожидаемый результат | На что смотрим |
|---|---|---|---|
| 1.1 | «Ну вот, короче, я думаю что нам нужно, ээ, переделать это всё» | «Я думаю, что нам нужно переделать это всё.» | Филлеры `ну вот`, `короче`, `ээ` убраны. Смысл сохранён |
| 1.2 | «Да» | «Да.» | Короткий ввод не расширяется и не дополняется |
| 1.3 | «Привет, это тестовое сообщение» | «Привет, это тестовое сообщение.» | Пунктуация расставлена, текст не изменён |
| 1.4 | «Я хотел сказать что это вот такая вот штука» | «Я хотел сказать, что это вот такая вот штука.» | `вот такая вот` сохранено — может быть манерой речи |
| 1.5 | «П-практичность очень важна» | «П-практичность очень важна.» | Заикание **сохранено** (ru:classic override) |
| 1.6 | «Мне нужно... ну типа... закончить проект» | «Мне нужно закончить проект.» | `ну типа` убрано, многоточие очищено |
| 1.7 | «Один два три четыре пять» | «Один, два, три, четыре, пять.» | Пунктуация при перечислении |
| 1.8 | (тишина / шум 1 сек) | Пустая строка или `""` | Не должно быть иероглифов, рандомных символов |

---

## 2. 💼 Деловой стиль (`business`)

> **Цель:** Чуть более формальный текст. Убираются речевые запинки, сглаживаются неловкие конструкции. Для писем, задач, рабочих чатов.

| # | Что говорим | Ожидаемый результат | На что смотрим |
|---|---|---|---|
| 2.1 | «Ну вот, слушай, я тут подумал что нам бы надо, это самое, созвон назначить» | «Я подумал, что нам нужно назначить созвон.» | Филлеры и запинки убраны, формулировка сглажена |
| 2.2 | «П-практичность очень важна» | «Практичность очень важна.» | Заикание **убрано** (ru:business override) |
| 2.3 | «Скинь мне пожалуйста отчет за вчера, ну и ещё бы хорошо если бы ты добавил графики» | «Скинь мне, пожалуйста, отчёт за вчера. Ещё было бы хорошо, если бы ты добавил графики.» | Разбивка на предложения, `ну` убрано |
| 2.4 | «Задача: проверить все API endpoints до конца недели» | «Задача: проверить все API endpoints до конца недели.» | Структурированный ввод не переписан |
| 2.5 | «Можно пожалуйста полностью убрать левое меню с проекта в клиенте» | «Можно, пожалуйста, полностью убрать левое меню с проекта в клиенте?» | Вопросительный знак, запятые |
| 2.6 | «Ну это, как бы, ну, ладно, давай созвонимся завтра» | «Давай созвонимся завтра.» | Вся «обвязка» убрана, ядро сохранено |
| 2.7 | «Надо также обсудить бюджет» | «Надо также обсудить бюджет.» | Слово `также` **сохранено** — не филлер |

---

## 3. 💻 Разработка (`tech`)

> **Цель:** Code-aware обработка. Технические термины нормализуются, код-диктовка конвертируется в каноничный код. Русская речь о технологиях обрабатывается с пониманием контекста.

| # | Что говорим | Ожидаемый результат | На что смотрим |
|---|---|---|---|
| 3.1 | «Нам нужно использовать юз стейт для хранения данных» | «Нам нужно использовать useState для хранения данных.» | `юз стейт` → `useState` |
| 3.2 | «Добавь консоль лог для юзер айди» | «Добавь console.log(userId).» | Код-диктовка конвертирована |
| 3.3 | «Напиши импорт реакт фром реакт» | «Напиши import React from "react".» | Полная code-строка восстановлена |
| 3.4 | «Запусти гит статус потом гит пуш origin main» | «Запусти git status, потом git push origin main.» | Команды терминала нормализованы |
| 3.5 | «Открой src слеш components слеш app точка tsx» | «Открой src/components/App.tsx.» | `слеш` → `/`, `точка` → `.` |
| 3.6 | «Бэкенд уже обрабатывает любое имя модели с транскрайб корректно» | «Бэкенд уже обрабатывает любое имя модели с transcribe корректно.» | Рассказ _о_ коде — остаётся прозой |
| 3.7 | «Это реакт компонент, он использует тайпскрипт» | «Это React-компонент, он использует TypeScript.» | Технические имена нормализованы |
| 3.8 | «Напиши конст юзер айди равно джейсон точка парс открывающая скобка response точка data закрывающая скобка» | «Напиши const userId = JSON.parse(response.data).» | Полная диктовка кода восстановлена |
| 3.9 | «Сделай карго чек и потом бан ран дев» | «Сделай cargo check и потом bun run dev.» | `карго чек` → `cargo check` |
| 3.10 | «Проблема в файле эй ай точка ар эс» | «Проблема в файле ai.rs.» | Расшифровка побуквенного произношения |

## 4. 🔇 Краевые случаи (все стили)

| # | Сценарий | Что говорим / делаем | Ожидаемый результат | На что смотрим |
|---|---|---|---|---|
| 4.1 | Очень короткая запись | «Да» (0.5 сек) | «Да.» | Не должно быть иероглифов |
| 4.2 | Шум без речи | (фоновый шум 2 сек) | `""` или ничего | Нет галлюцинаций STT |
| 4.3 | Смешанный язык | «Мне нужен API endpoint для user management» | «Мне нужен API endpoint для user management.» | Англицизмы не переводятся |
| 4.4 | Числа | «Двадцать пять тысяч рублей» | «25 000 рублей.» или «Двадцать пять тысяч рублей.» | Оба варианта допустимы |
| 4.5 | URL диктовка | «Перейди на proxy точка talkis точка ru» | «Перейди на proxy.talkis.ru.» | URL собран |
| 4.6 | Повтор слов | «Я я хочу хочу сказать» | «Я хочу сказать.» | Случайные повторы убраны |
| 4.7 | Длинный текст (20+ сек) | Диктовка абзаца текста | Полный абзац, корректная пунктуация | Текст не обрезан |
| 4.8 | Emoji/спецсимволы в речи | «Поставь звёздочку и плюсик» | Зависит от стиля | Нет крашей |
